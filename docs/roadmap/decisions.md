# Architecture Decisions

Durable engineering decisions extracted from the project's phase-by-phase development record, organized by topic rather than by when they happened (see [Project History](history.md) for the chronological account). These are the decisions still binding on new work today; the process/approval narrative each one originally came wrapped in has been left out.

## Genetics & Development

- **A genome is exactly three CPPNs** (`brain_cppn`, `morph_cppn`, `regulatory_cppn`) plus nothing else structural. `morph_cppn` is crossed/mutated/included in speciation distance but not read at growth time — disclosed technical debt, not an oversight.
- **Body-plan identity is a decoded combinatorial code, never a stored sequence.** `regulatory_cppn` generates a small recurrent network; running it produces gene states; three Hox-role genes, thresholded at 0.5, form a 3-bit code decoded via a fixed lookup table. This replaced an earlier literal `HoxSequence`/`HoxGene` design specifically because a stored, hand-authored sequence can't be smoothly mutated or evolved the way a decoded network can.
- **Development runs a fixed step count, never "to convergence."** Determinism requires this; an iterate-until-stable design would make the decode's cost and value depend on execution-order-sensitive convergence behavior.
- **Pigmentation is fully emergent**, decoded per-segment from three regulatory genes, never stored as RGB on the genome. `Diet::standard_color()` remains the fallback/reference palette for diet-based UI elements (charts, legends), not a claim about what any individual organism renders as.
- **Regional brains and reaction-diffusion morphogens are both real, implemented architecture** — not proposals. Regional-brain wiring metadata (`RegionId`) is deliberately CPU-side-only, never a GPU buffer/shader change, so it costs nothing when dormant (which, as of this writing, it always is — no genome has been observed to decode a `Ganglion` segment). Reaction-diffusion morphogens split cleanly: intra-organism signaling reuses the existing graph-relaxation pattern already proven by nutrient transport and hormone diffusion (the body is a ~15-node graph, not a continuous tissue mesh — this is architecturally a graph problem, not a spatial-PDE problem), while inter-organism/environmental coupling is a real, narrowly-scoped fifth GPU diffusion layer.
- **Schema-version-bump-no-migration is the standing policy for breaking genome/save changes.** This project does not build migration tooling for old save files when a breaking change is made; it bumps the version and accepts that old files stop loading. This has happened more than once and is expected to happen again.
- **Measure actuatable-effector rate directly before trusting any regulatory-seed or mutation-rate change.** This is not a general engineering platitude here — it is the specific, repeatedly-learned lesson from two real incidents (Phase 5's discovery that the original seed CPPN structurally couldn't reach the `Muscle` segment type, and Phase 9's discovery that founder-population mutation dosage was independently collapsing the same rate). A change to either area that isn't backed by a real headless measurement of the resulting population's effector rate should not be trusted.

## Neural Systems

- **Dual-CPPN separation (brain vs. body) avoids "broken brain" syndrome** — mutating the body plan independently of the brain topology is only safe because they're separately-evolved, separately-wired networks queried through stable interfaces (position-pair queries for the brain, position-index queries for the body).
- **CTRNN node states are clamped to `[-10.0, 10.0]`** during GPU integration specifically to prevent unbounded positive feedback from producing `NaN`s or extreme saturation — this clamp is load-bearing, not a leftover safety margin.
- **The current sensory input vector is 15 values** (3 scalar + 9-value binned vision), not the smaller vector used in some older test fixtures and (until this pass) some documentation — vision was rebuilt as part of the Phase 8 3D migration into an azimuth × elevation cone around the organism's 3D orientation frame.

## Camera, Rendering & 3D Engine

- **`Camera3d` is the single source of truth for camera/projection math**, full stop. Before this was consolidated, six separate places independently derived a projection matrix and three separately implemented screen↔world unprojection — all drifted out of sync. Any new code that needs camera math calls `Camera3d`'s own methods; it does not re-derive them.
- **Camera controllers (orbit, fly) only ever write into `Camera3d`** — they hold input-handling state and never duplicate its projection logic.
- **Organism rendering is mesh-based capsule instancing with physically-based shading**, replacing an earlier 2D SDF-metaball technique. This was treated as a real visual-identity change requiring explicit sign-off before implementation, not a routine refactor — alternatives (raymarched SDF, a rasterize-and-post-blend hybrid) were evaluated and explicitly rejected/deferred with stated reasons, not silently dropped.
- **`gpu` is compute-only.** Zero rendering-pipeline code exists in that crate, confirmed by direct code audit. This boundary is intentional and has held since before the 3D migration.
- **Chemical diffusion fields stay 2D world-space planes**, deliberately, even though physics and rendering are now 3D — a volumetric field would cost roughly two orders of magnitude more GPU memory/bandwidth than the current 2D layers, for a benefit that hasn't been measured as necessary.
- **The GPU physics broad-phase is a spatial hash, not a dense 3D grid** — a dense grid at 3D resolution matching the original 2D grid's density would need roughly 128× the memory.
- **Bilateral symmetry uses a body-fixed forward/dorsal frame** (`bilateral_fin_direction = dorsal × forward`), not a 2D perpendicular-vector trick. Radial symmetry was explicitly named and deferred as a separate, future body-plan hypothesis — not bundled into this change.
- **The CPU spatial index (`Octree`) and the GPU physics broad-phase (a fixed-size hash) are deliberately separate implementations**, not a shared generic abstraction — a 2D-structure-generalized-to-3D and a GPU-resident hash table have different enough constraints that forcing a shared interface would have made both worse. The same principle applies to the (never built) idea of unifying the 2D and (if ever built) 3D field/shape/broad-phase representations — build parallel implementations, don't force a shared abstraction across fundamentally different constraints.

## Ecosystem & Ecology

- **`EcologyConfig::max_organisms` must track the simulation's actual intended population scale.** This config field existed disconnected from `SimulationConfig::target_organism_count` (a separate, unread field) for long enough that it silently blocked all reproduction in every default-config run — the founder population, spawned directly and bypassing the cap check, always exceeded the stale default. Any future population-scale config should be wired through one path, not two independent numbers that happen to both mean "how many organisms."
- **`DeathCause` is exhaustively matched everywhere it's consumed, even for variants no code path constructs yet** (`GodMode`, `Injury`, `Environment` were unconstructed for a long stretch of this project's history) — a future system that starts producing one of these needs no rendering/event-handling change, only a new construction site.

## UI & Interaction

- **One selection/follow mutation pathway.** `WorkbenchState::select`/`set_follow` are the only legitimate way to change selection/tracking state, with a narrow, named exception for plain tab-navigation. This is a standing rule, not a one-time cleanup.
- **A four-tier event-communication model**, decided by one test question ("if a researcher reopened this experiment a week later, would they need this event to still exist somewhere?"): silent (continuous/high-frequency), local visual feedback (world-anchored, tick-expiring), session notification (scrollable log entries and ephemeral toasts — two distinct sub-cases, not to be conflated), and persistent research event (durable, exportable, part of the scientific record).
- **A shared graph-canvas primitive is HOW-only, never WHAT.** The Neural Viewer and GRN Viewer share one generic node-link-drawing primitive with zero domain knowledge; each viewer keeps its own layout algorithm, node classification, and color scheme. Two near-identical color-token pairs (`SYNAPSE_*` vs. `EDGE_*`) are kept separately named on purpose, not merged, since they belong to conceptually distinct domains that happen to reuse the same RGB values today.
- **Workspace layouts have one canonical shape** (`panel_modes` + `layout_shares`) whether they're a built-in preset or a user-saved layout — and any imported layout is sanitized (unknown panel names dropped, non-finite/non-positive shares replaced) before use.
- **The tick loop is a hand-written, fixed-order function, not a generic scheduler.** A `scheduler` crate exists in the workspace but is not used by the running application — it's retained only as a benchmark fixture and integration-test target. Don't describe the live app as "using a scheduler."
- **`Local<T>` cannot hold cross-tick state** under this app's actual driving pattern (`RunSystemOnce` builds a fresh `SystemState`, and therefore a fresh `Local`, on every call) — any system that needs to remember something between ticks must use a `Resource`. This exact mistake has caused at least two real, silent bugs in this project's history (a diagnostic module that never advanced past its first tick; a hazard-lifecycle system whose elapsed-time tracking was always zero).

## Research, Persistence & Process

- **`SimRng` (seeded `ChaCha8Rng`) is the only sanctioned source of randomness** for anything that affects simulation outcomes. Direct use of an unseeded RNG in a simulation-affecting path is a determinism bug, found and fixed more than once in this project's history — see [Determinism](../explanation/determinism.md) for what is and isn't actually guaranteed as a result.
- **The bridge-module pattern**: every module that touches both the ECS world and a "pure" supporting crate (`analytics`, `research`, `storage`) lives in `app`, the composition root. Pure crates never depend on `bevy_ecs`-touching crates directly.
- **`app` is the only crate permitted to depend on everything else** — a convention, not (yet) compiler-enforced.
